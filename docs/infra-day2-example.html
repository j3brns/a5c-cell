Worked example

Use Symphonic to run infrastructure work from an issue bank.

A new platform repo can use Terraform for cloud resources, Helm charts for Kubernetes packaging, and Argo CD for GitOps deployment. Symphonic sits above those tools: it turns planned infrastructure work into isolated agent runs, gates the change, and preserves Day 2 evidence.

Terraform Helm Argo CD GitLab issues Git worktrees

New codebase bootstrap

Start by making the repo runnable by an agent and boring to validate.

1. Add repo contract

Add WORKFLOW.md, AGENTS.md, issue template, schema, and a Makefile. The workflow points at the GitLab issue bank and sets workspace.mode: git-worktree.

2. Add infra gates

Make Terraform, Helm, YAML, and shell checks deterministic. Do not let an agent decide which checks count after the work has already changed.

3. Generate tasks

Use the plan generator to create schema-shaped issue bodies, review them, then create GitLab issues. Symphonic only launches issues that meet Definition of Start. 

PYTHONPATH=src python3 -m symphonic.cli generate-tasks platform-plan.yaml --start-seq 2000
glab issue create -R group/platform-live --title "TASK-2000: Bootstrap remote state" --description-file task-2000.md
make validate-issues
make pilot
make run-once

Symphonic workflow excerpt

tracker:
  kind: gitlab
  project_slug: group/platform-live
  active_labels: ["status:not-started", "status:in-progress"]
workspace:
  root: ~/.cache/symphonic/workspaces
  mode: git-worktree
  base_ref: main
agent:
  profile: codex
  max_concurrent_agents: 2
quality_gates:
  default_mode: infra-fast
  modes:
    infra-fast:
      - label: Terraform and Helm validation
        target: validate-local
        command: "make validate-local"
      - label: GitOps diff
        target: gitops-diff
        command: "make gitops-diff"
Worked issue flow

Example: add a new service through Terraform, Helm, and Argo CD.

Terraform task

Acceptance criteria include a reviewed plan artifact, no unplanned replacement, provider version pinning, and rollback notes. The agent can edit modules but cannot apply changes.

Helm task

The gate renders manifests and validates Kubernetes schemas before review. UI/service changes add smoke tests or Playwright checks when relevant.

Argo CD task

The MR includes app diff output, sync-wave rationale, health expectations, and rollback route. Sync remains an operator or pipeline action, not an agent side effect.

Day 2 operations

Symphonic turns operational signals into controlled repair loops.

Drift

A scheduled drift check opens an issue with the plan output, suspected cause, and required validation. Symphonic runs the repair in a worktree and leaves a receipt.

Upgrade

Kubernetes, provider, chart, and module upgrades become sequenced issues with explicit dependency edges and rollback gates.

Incident follow-up

Post-incident actions become tasks with impact, evidence, tests, and docs. The issue bank prevents fixes from staying in chat or a meeting note.

Compliance

Policy changes, IAM tightening, and audit evidence are gated like code. The agent can draft changes, but the gate records proof before handoff.

Day 2 operating rhythm

  1. Signals arrive from Argo CD health, cloud audit findings, drift detection, vulnerability feeds, or service SLOs.
  2. The platform operator converts the signal into a GitLab issue with `Seq:`, `Risk:`, dependencies, acceptance criteria, test plan, docs plan, and review plan.
  3. Symphonic selects the next runnable issue, creates a worktree, prompts the chosen agent, and runs the infra gate.
  4. The merge request carries Terraform plan output, rendered manifests, Argo CD diff, quality-gate receipts, and rollback notes.
  5. After merge and sync, the issue gets handoff evidence. Any new signal becomes the next issue.
Practical boundary

Symphonic does not replace Terraform, Helm, Argo CD, CI, or human production approval. It coordinates agent work around them: issue selection, isolated runtime, deterministic gates, evidence, and Day 2 continuity.
